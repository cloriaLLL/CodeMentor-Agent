# CodeMentor Agent 完美顶层架构与工程规范 (Perfect Architecture Spec)

## 1. 系统概述与核心目标 (System Overview)

[cite_start]CodeMentor Agent 是一款工程导向、按“学 → 练 → 用”标准化节拍运行的计算机技术学习 Agent 系统 [cite: 1, 44, 49][cite_start]。系统旨在消除传统 LLM 泛答的非结构化与幻觉问题，同时解决纯算法刷题（如 LeetCode）脱离工业实际应用场景的痛点 [cite: 1, 31, 44]。

### 1.1 核心设计原则
1. [cite_start]**节拍驱动 (Rhythm-Driven)**：由 Orchestrator 状态机按“概念与演进 → 示例 → 场景 Mini-Project 练习 → 工业生态收尾”的主动推演逻辑运行 [cite: 1, 45, 91]。
2. [cite_start]**严格基准 (Strict Grounding)**：RAG 召回机制强制依赖官方 Docs / PEP 规范，从根源杜绝模型幻觉 [cite: 30, 31]。
3. [cite_start]**零坏题率 (Zero-Broken Guarantee)**：采用出题 Generator 与校验 Validator 双 Agent 预跑机制，确保推送给用户的代码题目与单元测试用例 100% 可通过 [cite: 32, 91]。
4. [cite_start]**Schema 驱动扩展 (Schema-Driven Multi-Language Engine)**：核心 Agent 业务逻辑与特定编程语言解耦，通过配置文件无缝扩展技术栈 [cite: 33, 53]。

## 2. 系统整体架构与组件定义 (System Architecture)

系统采用模块化 Agentic Workflow 架构，包含四大核心服务组件：


```

+-----------------------------------------------------------------------------------+
|                                Front-End Client                                   |
|             (Chat Workspace UI + Split-Screen Code Editor Workspace)             |
+-----------------------------------------+-----------------------------------------+
| WebSocket / HTTP REST API
v
+-----------------------------------------------------------------------------------+
|                            Orchestrator Service (主控)                            |
|                 - State Machine Engine (Teach / Practice / Ecosystem)             |
|                 - Learner Profile Context Engine                                  |
+--------------+--------------------------+--------------------------+--------------+
|                          |                          |
v                          v                          v
+------------------------------+ +------------------------+ +-----------------------+
|     RAG Ingestion & Vector   | |  Dual-Agent Practice   | | Isolated Execution    |
|            Engine            | |        Engine          | |        Sandbox        |
| - Offical Docs / PEP Storage | | - Generator Agent      | | - Test Runner Engine  |
| - Metadata Chunking & Doc2Q  | | - Validator Agent      | | - AST Code Inspector  |
| - Strict Grounding Filter    | | - Auto-Retry Loop      | | - Execution Result  |
+------------------------------+ +------------------------+ +-----------------------+

```

### 2.1 组件职责定义

* **Orchestrator Service (教学状态主控)**：
  * [cite_start]管理当前学习会话的状态（`TeachMode` / `PracticeMode` / `EcosystemMode`）[cite: 91]。
  * [cite_start]读取并更新用户画像上下文（`learner_profile`），控制节点推演图谱 [cite: 2, 4]。
* **RAG Ingestion & Vector Engine (知识检索引擎)**：
  * [cite_start]解析并管理官方合规源（PEP、Official Specs、RFC 等）[cite: 3, 30]。
  * [cite_start]按知识元（Knowledge Atom）颗粒度切割，带有版本演进与依赖节点等 Metadata 标记 [cite: 30]。
* **Dual-Agent Practice Engine (双 Agent 出题与校验引擎)**：
  * [cite_start]**Generator Agent**：基于 AST 提取的开源种子代码，按当前节点生成场景化 Mini-Project 描述与单元测试 [cite: 12, 32, 51]。
  * [cite_start]**Validator Agent**：在题目推送到前端前，静默调用沙盒预跑参考答案，不通过则捕获 Trace 触发内部重试 Loop [cite: 32, 52]。
* **Isolated Execution Sandbox (沙盒执行环境)**：
  * [cite_start]隔离执行用户代码或参考代码，运行内嵌的单元测试框架（如 pytest / go test）[cite: 52, 85]。
  * [cite_start]返回结构化的运行 stdout/stderr、测试结果断言、耗时与内存开销 [cite: 85, 88]。

## 5. 跨语言扩展 Schema 引擎设计 (Extension Engine)

为了让系统支持 Go、Rust 等语言扩展，定义 `language_manifest.yaml` 标准插件契约：

```yaml
language: go
display_name: "Go Language"
doc_ingestion:
  source_type: "markdown_spec"
  index_path: "docs/go_spec/"
  chunk_strategy: "by_node_id"
sandbox_config:
  runner_image: "golang:1.22-alpine"
  test_command: "go test -v ./..."
  file_extension: ".go"
seed_pool_path: "seeds/go/"

```

Agent 系统在启动时加载特定语言的 `language_manifest.yaml`，即刻切换出题和判题上下文，无需硬编码多语言分支逻辑 。

```
