# 对话存储架构重构计划

## Context（背景）

当前项目存在三个架构短板：

1. **对话无状态**：后端 `OrchestratorAgent` 不持久化对话历史，每次请求由前端传完整 `history`。无法实现"每个对话 = 一个大模型对话"的服务端管理。
2. **笔记本无后端支撑**：笔记本（含章节）仅存在前端 localStorage，无父级总结机制，无法模拟 AI 网站的"笔记本/分组"功能。
3. **练习出题无记忆**：`ExerciseGeneratorAgent` 每次独立调用 LLM，不知道之前出过什么题，导致重复出题、无法持续进阶。

本次重构在后端 SQLite 新增对话存储层，统一管理三种场景的对话历史，实现：服务端对话管理、笔记本父级摘要、练习模块按"语言+题型"分类的对话存储与清除。

## 数据库 Schema（2 张新表）

在 [database.py](file:///c:/WorkSpace/CodeMentor%20Agent/app/core/database.py) 的 `_SCHEMA` 末尾追加：

```sql
-- 对话元数据
CREATE TABLE IF NOT EXISTS conversations (
    conversation_id  TEXT PRIMARY KEY,
    user_id          TEXT DEFAULT 'default',
    type             TEXT NOT NULL,        -- chat | notebook_parent | notebook_chapter | exercise_module
    title            TEXT DEFAULT '',
    parent_id        TEXT,                  -- notebook_chapter 指向 notebook_parent
    module_key       TEXT,                  -- exercise_module 专用，如 'python:understanding:choice'
    summary          TEXT DEFAULT '',       -- notebook_parent 的累积摘要
    meta             TEXT DEFAULT '{}',
    message_count    INTEGER DEFAULT 0,
    created_at       TEXT DEFAULT (datetime('now')),
    updated_at       TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (parent_id) REFERENCES conversations(conversation_id) ON DELETE SET NULL
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_conv_module_unique ON conversations(module_key) WHERE module_key IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_conv_parent ON conversations(parent_id);

-- 对话消息
CREATE TABLE IF NOT EXISTS conversation_messages (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id  TEXT NOT NULL,
    role             TEXT NOT NULL,         -- user | assistant
    content          TEXT NOT NULL,
    msg_meta         TEXT DEFAULT '{}',
    seq              INTEGER NOT NULL,      -- 同对话内自增序号
    created_at       TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (conversation_id) REFERENCES conversations(conversation_id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_msg_conv ON conversation_messages(conversation_id, seq);
```

## 实现步骤（6 个阶段）

### 阶段 1：存储层（无依赖）

**新建** [app/services/conversation_store.py](file:///c:/WorkSpace/CodeMentor%20Agent/app/services/conversation_store.py)

`ConversationStore` 类，核心方法：

| 方法 | 用途 |
|------|------|
| `get_or_create(conversation_id, type, parent_id, module_key, meta)` | 幂等获取/创建对话 |
| `get_or_create_module_conversation(module_key, meta)` | 按 module_key 获取/创建练习模块对话（唯一约束） |
| `add_message(conversation_id, role, content, msg_meta)` | 追加消息，自动计算 seq，递增 message_count |
| `get_messages_as_llm_history(conversation_id, limit=20)` | 返回 `[{role, content}, ...]` 供 LLM 调用 |
| `clear_module_conversation(module_key)` | 清除模块消息（保留行，重置 count/summary） |
| `get_parent_summary(parent_id)` | 读取笔记本父级摘要 |
| `refresh_parent_summary(parent_id, llm_provider)` | 汇总各章节消息 → LLM 生成摘要 → 存入父级 |
| `should_refresh_summary(parent_id, threshold=6)` | 判断是否需要刷新（避免频繁 LLM 调用） |

关键实现细节：
- `add_message` 中 seq 用单语句 `INSERT ... SELECT COALESCE(MAX(seq),0)+1` 避免竞态
- `get_or_create_module_conversation` 用 `INSERT OR IGNORE` + `SELECT` 处理并发
- `refresh_parent_summary` 构建各章节最近消息的摘要 prompt，调用 LLM 生成 ≤500 字总结

## 向后兼容保障

- `ChatRequest.conversation_id=None` → 后端走 `req.history` 路径，行为不变
- `ExerciseGenerateResponse` 新增字段均为 `Optional`，旧前端自动忽略
- 数据库 `CREATE TABLE IF NOT EXISTS` 幂等，不影响已有表
- Agent 新增参数默认 `None`，不传时行为完全不变
- 前端旧 localStorage 数据无 `conversationId` → 首次发消息时后端 `get_or_create` 自动补建

## 验证方案

1. **后端单元验证**：启动后端，用 curl 测试：
   - `POST /api/chat_stream` 带 `conversation_id` → 验证消息持久化到 SQLite
   - `POST /api/exercise/generate` 连续两次同模块 → 验证第二次 LLM 收到历史上下文
   - `POST /api/exercise/clear-module` → 验证消息清除但对话行保留
2. **前端集成验证**：
   - 新建对话 → 发送消息 → 刷新页面 → 继续发送 → 验证后端历史加载正常
   - 新建笔记本 → 在章节1发消息 → 切换到章节2发消息 → 验证章节2收到章节1的父级摘要
   - 练习启动器 → 连续生成同类型题目 → 验证题目不重复 → 点击"清除出题记忆" → 验证下次生成如同全新
3. **兼容性验证**：不传 `conversation_id` 的旧请求 → 验证走 history 路径正常
4. **TypeScript 类型检查**：`npx tsc --noEmit` 无错误
5. **Lint 检查**：`npm run lint` 无新增错误
