# P1 架构重构实施方案

## Context（背景与目标）

前期全面诊断发现 `agents/` 模块存在三大架构性技术债：

1. **God Class**：`agents/exercise_evaluator.py` 单文件 922 行、`ExerciseEvaluator` 单类 16 方法承担 5 项职责（路由/理解题判分/沙箱执行/代码质量评估/AI反馈生成），圈复杂度爆炸且零单元测试。
2. **架构分裂**：新旧两套 Generator 管线并存且不互通——旧链路（`practice.py`→`validator.py`→`generator.py`，5字段 `GeneratedExercise`）与新链路（`exercise.py`→`exercise_service.py`→`exercise_generator.py`+`exercise_evaluator.py`，17字段 `GeneratedExercise`）。经核查，**前端对旧链路零引用**，旧链路是死代码。
3. **Prompt 内嵌**：~500 行 prompt 字符串硬编码于源码（`MENTOR_SYSTEM_PROMPT` 250行 + 4个题型prompt + 2个反馈prompt），违反 `load_prompt()` 既有约定，调参与 A/B 测试困难。

目标：通过三个独立可发布的子任务消除上述技术债，提升可维护性与扩展性，且**零功能回归**。

## 执行顺序

**B（清理旧链路）→ A（拆分 God Class）→ C（外置 prompt）**

理由：B 先行可减少 A/C 阶段需处理的代码量；旧 `GeneratedExercise` 与新 `GeneratedExercise` 同名易混淆，先消除；A 完成后 evaluator 的 `_build_*_system_prompt` 集中在单一文件，便于 C 阶段外置。

## 子任务 A：拆分 ExerciseEvaluator 为子包

### 新建文件（采用 `agents/language_service/` 子包先例）

```
agents/exercise_evaluator/
├── __init__.py       # re-export 公开 API
├── models.py         # EvaluationResult Pydantic 模型
├── quality.py        # CodeQualityAssessor（6个 _assess_* 纯函数）
├── ai_feedback.py    # AIFeedbackBuilder（5个 LLM 反馈方法）
└── core.py           # 瘦身后的 ExerciseEvaluator（路由+沙箱编排，~350行）
```

### 拆分边界（基于方法聚类）

| 新类 | 迁入方法（原行号） | 依赖 |
|---|---|---|
| `CodeQualityAssessor` | `_assess_code_quality`→`assess_code_quality`, `_assess_code_quality_python`, `_assess_code_quality_generic`, `_assess_structure`→`assess_structure`, `_assess_structure_generic`, `_assess_boundary_handling`→`assess_boundary_handling` | 纯函数，仅 ast/re |
| `AIFeedbackBuilder` | `_generate_understanding_ai_feedback`→`generate_understanding_feedback`, `_build_understanding_feedback_system_prompt`, `_generate_ai_feedback`→`generate_code_feedback`, `_build_ai_feedback_system_prompt`, `_build_ai_feedback_user_message` | self.llm, self._llm_enabled |
| `ExerciseEvaluator`（瘦身） | `__init__`, `evaluate`, `_evaluate_understanding`, `_evaluate_code`, `_evaluate_project` | 组合上述两类 + sandbox |

### 接口设计（向后兼容）

```python
# core.py — ExerciseEvaluator 入口签名不变
class ExerciseEvaluator:
    def __init__(self, llm_provider: Optional[LLMProvider] = None) -> None:
        self._quality = CodeQualityAssessor()
        self._feedback = AIFeedbackBuilder(llm_provider)

    @property
    def llm(self): return self._feedback.llm           # 向后兼容
    @property
    def _llm_enabled(self) -> bool: return self._feedback._llm_enabled

    async def evaluate(self, exercise, user_answer) -> EvaluationResult: ...  # 签名不变
```

- `app/services/exercise_service.py:11,30,118` **无需改动**（import 路径 `agents.exercise_evaluator` + 类名 + 方法签名均不变）
- 删除原 `agents/exercise_evaluator.py`（与子包同名，必须删除否则 import 歧义）

### 修改文件
- `agents/orchestrator.py`：删除 `MENTOR_SYSTEM_PROMPT`/`NOTEBOOK_MODE_SUFFIX` 常量，`__init__` 改 `self.SYSTEM_PROMPT = load_prompt("mentor")`，`_build_messages` 改 `load_prompt("mentor_notebook_suffix")`

### 关键风险点
- **`{{` `}}` 转义**：`exercise_*.txt` 中 JSON 示例的 `{{` `}}` 必须保留（`.format()` 会解析为字面 `{` `}`）。外置后对每个模板跑 `.format(...)` 冒烟
- **首尾空白一致性**：`mentor.txt` 首行换行需与原字符串一致，迁移时用 `assert MENTOR_SYSTEM_PROMPT == load_prompt("mentor")` 验证后再删常量
- **文件编码**：必须 UTF-8（`load_prompt` 已显式 `read_text(encoding="utf-8")`）
